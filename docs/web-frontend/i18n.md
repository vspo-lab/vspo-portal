# Internationalization (i18n)

## Configuration

| Aspect | Value |
|--------|-------|
| Library | next-i18next (wrapping i18next + react-i18next) |
| next-i18next defaultLocale | `"default"` (pseudo-locale; actual default is `ja` via middleware) |
| Frontend locales | `default`, `en`, `ja`, `cn`, `tw`, `ko` |
| Detection | Disabled in next-i18next (`localeDetection: false`); handled by custom middleware |
| Translation files | `public/locales/{locale}/{namespace}.json` |
| Default namespace | `common` |

## Locale Resolution

Handled by `src/middleware.ts` with this priority:

1. URL path prefix (`/en/schedule/all`)
2. `NEXT_LOCALE` cookie
3. `Accept-Language` header
4. Default: `ja`

The middleware sets the `NEXT_LOCALE` cookie on every request, ensuring consistent locale across navigation.

## Server-Side Translations

Dual-environment support to work on both Cloudflare Workers and Node.js:

### Cloudflare Edge

Uses a custom `CloudflareAssetsBackend` (`lib/i18n/cf-assets.ts`) that implements the i18next `BackendModule` interface:

- Fetches `/locales/{lng}/{ns}.json` from Cloudflare `ASSETS` binding
- Validates ASSETS binding availability via Cloudflare context detection
- Wraps all operations in Result type (no try-catch)
- Handles 404s and parse failures gracefully

### Node.js (Local Dev)

Uses standard `next-i18next/serverSideTranslations` reading from the filesystem.

### Unified Loader (`lib/i18n/server.ts`)

Auto-detects the environment and chooses the appropriate loading strategy:

```typescript
// Cloudflare: creates i18n instance with CloudflareAssetsBackend
// Node.js: delegates to next-i18next/serverSideTranslations
// Returns empty translations on failure (no hard errors)
```

### Loading in serverSideProps

```tsx
// features/<feature>/pages/<Page>/serverSideProps.ts
const translations = await serverSideTranslations(locale, ["common", "streams"]);
return { props: { ...translations } };
```

## Client-Side Usage

Use `useTranslation` from `next-i18next` with namespace(s) matching the JSON filenames under `public/locales/{locale}/`:

```tsx
const { t } = useTranslation("common");        // single namespace
const { t } = useTranslation(["common", "streams"]); // multiple
```

Each feature has a corresponding namespace (e.g., `streams`, `clips`, `freechat`, `multiview`, `site-news`).

## Adding a New Locale

1. Add the locale code to `next-i18next.config.js` `locales` array
2. Add locale to middleware's supported locales list
3. Add locale-to-timezone mapping in `@vspo-lab/dayjs` `LOCALE_TIMEZONE_MAP`
4. Create translation files under `public/locales/{new-locale}/`
5. Add date-fns locale import in `lib/utils.ts` `getLocale()`
