# Internationalization (i18n)

## Configuration

| Aspect | Value |
|--------|-------|
| Library | `next-intl` |
| Config | `src/i18n/routing.ts` |
| Locales | `en`, `ja`, `cn`, `tw`, `ko` |
| Default locale | `ja` |
| Locale prefix | `as-needed` (omitted for `ja`) |
| Translation files | `public/locales/{locale}/{namespace}.json` |
| Namespaces (12) | `about`, `clips`, `common`, `events`, `freechat`, `meta`, `multiview`, `privacy`, `schedule`, `site-news`, `streams`, `terms` |

## Routing Definition

`src/i18n/routing.ts` defines locales and prefix strategy:

```typescript
import { defineRouting } from "next-intl/routing";

export const routing = defineRouting({
  locales: ["en", "ja", "cn", "tw", "ko"],
  defaultLocale: "ja",
  localePrefix: "as-needed",
});
```

## Translation Loading

`src/i18n/request.ts` loads all 12 namespaces per request via `getRequestConfig`. Dual-environment support:

| Environment | Strategy |
|-------------|----------|
| Cloudflare Workers | `ASSETS.fetch()` binding fetches `/locales/{locale}/{ns}.json` |
| Local dev (Node.js) | Dynamic `import()` from filesystem |

All namespaces are loaded in parallel with `Promise.all` and merged into a single `messages` object keyed by namespace.

## Navigation

`src/i18n/navigation.ts` exports locale-aware navigation primitives:

```typescript
import { createNavigation } from "next-intl/navigation";
import { routing } from "./routing";

export const { Link, usePathname, useRouter } = createNavigation(routing);
```

Use these instead of Next.js built-in `Link` and `useRouter` to ensure locale-prefixed URLs.

## Server-Side Usage

Use `getTranslations` from `next-intl/server` in async Server Components and `generateMetadata`:

```typescript
import { getTranslations } from "next-intl/server";

// In generateMetadata or page body
const t = await getTranslations({ locale, namespace: "streams" });
const tCommon = await getTranslations({ locale, namespace: "common" });

const title = `${tCommon("spodule")} | ${t("titles.streamSchedule")}`;
```

For multiple namespaces, call `getTranslations` separately for each.

## Client-Side Usage

Use `useTranslations` from `next-intl` in Client Components:

```typescript
import { useTranslations } from "next-intl";

const t = useTranslations("common");     // single namespace
const tStreams = useTranslations("streams"); // another namespace
```

Note: `useTranslations` returns the `t` function directly (no destructuring). For multiple namespaces, use separate `useTranslations` calls.

## Middleware

`src/middleware.ts` wraps `next-intl/middleware` with additional cookie handling:

```typescript
import createIntlMiddleware from "next-intl/middleware";
import { routing } from "./i18n/routing";

const intlMiddleware = createIntlMiddleware(routing);

export default function middleware(request: NextRequest) {
  const response = intlMiddleware(request);
  setTimeZone(request, response);
  setSessionId(request, response);
  return response;
}
```

The `next-intl` middleware handles locale detection (URL path, `Accept-Language` header) and sets the `NEXT_LOCALE` cookie.

## Adding a New Locale

1. Add the locale code to `routing.locales` in `src/i18n/routing.ts`
2. Add locale-to-timezone mapping in `@vspo-lab/dayjs` `LOCALE_TIMEZONE_MAP`
3. Create translation files under `public/locales/{new-locale}/` for all 12 namespaces
4. Add `generateStaticParams` entry in SSG pages (about, privacy-policy, terms, site-news)
5. Add date-fns locale import in `lib/utils.ts` `getLocale()`
