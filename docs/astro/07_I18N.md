# i18n Improvements

## Current Implementation

### Astro Configuration

```typescript
// astro.config.ts
i18n: {
  defaultLocale: "ja",
  locales: ["ja", "en"],
  routing: "manual",
}
```

### Translation System

- `src/i18n/dict.ts` â€” ~420 lines. ja/en dictionary object + `t()` function
- `t(locale, key, params?)` â€” Retrieves translation text via dot-notation keys, supports `{param}` substitution
- Type-safe helpers: `memberTypeKey()`, `memberTypeDescKey()`, `languageDisplayKey()`

### Locale Switching

- `POST /api/change-locale` â€” Saves locale to session, page reload
- Called via form POST from LanguageSelector

## Issues

1. **URL-based routing not used** â€” Astro's i18n feature (`/en/dashboard` etc.) is unused. Session-based approach is SEO-unfriendly
2. **`routing: "manual"`** â€” Astro's automatic route generation is not leveraged
3. **Dictionary file bloat** â€” 420 lines in one file. No feature-based splitting
4. **Translation key type safety** â€” Key typos not detected until runtime
5. **Translations for React islands** â€” Current `t()` is server-side only. Cannot be used inside React islands

### Phase 3: Passing Translations to React Islands

#### Approach A: Via Props (Recommended)

```astro
---
const locale = getLocale(Astro);
const translations = {
  save: t(locale, "channel.save"),
  cancel: t(locale, "channel.cancel"),
  // ... only necessary translations
};
---
<ChannelConfigModal client:load translations={translations} />
```

**Pros**: Translations resolved server-side. No need to include entire dictionary in island bundle
**Cons**: Increasing prop count

#### Approach B: Via Nano Store

```typescript
// i18n/stores/locale.ts
import { atom } from "nanostores";

export const $locale = atom<"ja" | "en">("ja");
```

```tsx
// Inside React island
import { useStore } from "@nanostores/react";
import { $locale } from "../../i18n/stores/locale";
import { t } from "../../i18n";

function ChannelConfigModal() {
  const locale = useStore($locale);
  return <button>{t(locale, "channel.save")}</button>;
}
```

**Pros**: No props needed
**Cons**: Entire dictionary included in client bundle

#### Recommendation

Small islands â†’ **Approach A** (props)
Large islands (ChannelConfigModal etc.) â†’ **Approach B** (Nano Store) â€” too many translation keys for props to manage

### Phase 4: Browser Language Detection

#### Current State

The bot-dashboard uses session-based locale selection. Users manually switch via `LanguageSelector`. There is no automatic detection of the user's browser language preference.

#### Proposed: Use `Astro.preferredLocale`

Astro provides built-in browser language detection via the `Accept-Language` header:

```astro
---
// middleware.ts or any SSR page
const preferred = Astro.preferredLocale;
// e.g. "ja" if browser sends Accept-Language: ja,en;q=0.9
// undefined if no match against configured locales

const allMatches = Astro.preferredLocaleList;
// e.g. ["ja", "en"] â€” all locales matching both browser and site config

const current = Astro.currentLocale;
// e.g. "ja" â€” computed from URL, falls back to defaultLocale
---
```

| Property | Type | Description |
|----------|------|-------------|
| `Astro.preferredLocale` | `string \| undefined` | Best match between browser `Accept-Language` and configured `locales` |
| `Astro.preferredLocaleList` | `string[]` | All matching locales, ordered by browser preference |
| `Astro.currentLocale` | `string \| undefined` | Locale derived from URL path, defaults to `defaultLocale` |

**Note**: `preferredLocale` and `preferredLocaleList` are only available for on-demand rendered pages (SSR). They cannot be used on prerendered static pages.

#### Use Case: Auto-Detect on First Visit

```typescript
// middleware.ts
const localeMiddleware = defineMiddleware(async (context, next) => {
  const sessionLocale = context.locals.locale;

  // If no session locale set, detect from browser
  if (!sessionLocale) {
    const detected = context.preferredLocale ?? "ja";
    context.locals.locale = detected;
    // Optionally persist to session
    await context.session?.set("locale", detected);
  }

  return next();
});
```

### Phase 5: Leverage `astro:i18n` Middleware Helpers

#### Current State

The project uses `routing: "manual"`, which disables Astro's built-in i18n middleware. However, Astro provides helper functions from `astro:i18n` that can be used alongside custom middleware.

#### Available Helpers

```typescript
import { middleware, redirectToDefaultLocale, notFound, redirectToFallback } from "astro:i18n";
```

| Helper | Description |
|--------|-------------|
| `middleware(options)` | Creates Astro's i18n middleware with custom routing options |
| `redirectToDefaultLocale(status)` | Returns a redirect response to the default locale |
| `notFound()` | Returns a 404 response |
| `redirectToFallback()` | Returns a redirect to the configured fallback locale |

#### Proposed: Compose with Custom Middleware

Instead of replacing Astro's i18n middleware entirely, compose it with custom logic using `sequence`:

```typescript
// middleware.ts
import { defineMiddleware, sequence } from "astro:middleware";
import { middleware as i18nMiddleware } from "astro:i18n";

const customLocaleMiddleware = defineMiddleware(async (context, next) => {
  // Custom locale logic (session-based, cookie-based, etc.)
  const response = await next();

  // Handle specific routes differently
  if (context.url.pathname.startsWith("/dashboard")) {
    // Dashboard routes use session-based locale
    return response;
  }

  return response;
});

export const onRequest = sequence(
  securityHeaders,
  auth,
  customLocaleMiddleware,
  i18nMiddleware({
    redirectToDefaultLocale: false,
    prefixDefaultLocale: false,
  }),
);
```

#### Benefits

- Reuses Astro's tested routing logic for standard paths
- Custom middleware handles app-specific cases (dashboard, API routes)
- `redirectToDefaultLocale: false` avoids infinite redirect loops when `prefixDefaultLocale` is also false

### Phase 6: URL-Based Routing (Future)

Currently session-based (`routing: "manual"`). Migration to URL-based (`/ja/dashboard`, `/en/dashboard`) is a major change, to be considered in a separate phase.

**Migration benefits**:

- SEO: Each language version has its own URL â†’ Google indexing works properly
- Sharing: Locale information embedded in URL
- SSR: No session needed, cacheable

**Migration cost**:

- All routing changes
- Middleware changes
- Redirect rule additions
- Existing bookmark compatibility

```typescript
// Future astro.config.ts
i18n: {
  defaultLocale: "ja",
  locales: ["ja", "en"],
  routing: {
    prefixDefaultLocale: false, // /dashboard = ja, /en/dashboard = en
  },
}
```
