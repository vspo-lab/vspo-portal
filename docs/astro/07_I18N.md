# i18n 改善

## 現状の実装

### Astro 設定

```typescript
// astro.config.ts
i18n: {
  defaultLocale: "ja",
  locales: ["ja", "en"],
  routing: "manual",
}
```

### 翻訳システム

- `src/i18n/dict.ts` — ~420行。ja/en の辞書オブジェクト + `t()` 関数
- `t(locale, key, params?)` — ドット記法のキーで翻訳テキスト取得、`{param}` 置換対応
- 型安全な helper: `memberTypeKey()`, `memberTypeDescKey()`, `languageDisplayKey()`

### ロケール切替

- `POST /api/change-locale` — セッションにロケール保存、ページリロード
- LanguageSelector からフォーム POST で呼び出し

## 問題点

1. **URL ベースルーティング未使用** — Astro の i18n 機能 (`/en/dashboard` 等) を使っていない。セッションベースのため SEO に不利
2. **`routing: "manual"`** — Astro の自動ルーティング生成を活用していない
3. **辞書ファイルの肥大化** — 420行が1ファイル。機能ごとの分割なし
4. **翻訳キーの型安全性** — キーの typo が実行時まで検出されない
5. **React island への翻訳渡し** — 現在の `t()` はサーバーサイドで使用。React island 内では使えない

## 改善案

### Phase 1: 辞書ファイルの構造化

```text
src/i18n/
  dict.ts              → index.ts (re-export)
  locales/
    ja/
      common.ts        # 共通テキスト (header, footer, buttons)
      landing.ts       # LP テキスト
      dashboard.ts     # ダッシュボードテキスト
      channel.ts       # チャンネル設定テキスト
      auth.ts          # 認証テキスト
    en/
      common.ts
      landing.ts
      dashboard.ts
      channel.ts
      auth.ts
```

**メリット**:

- 機能ごとの翻訳管理
- 翻訳漏れの検出が容易
- PR レビューが機能単位で可能

### Phase 2: 型安全性の強化

```typescript
// i18n/types.ts
// ja をソースとして en の型を推論
import type { jaCommon } from "./locales/ja/common";

type TranslationKeys = FlattenKeys<typeof jaCommon>;
// "header.title" | "header.logout" | "footer.copyright" | ...

// t() 関数の型を強化
function t(locale: Locale, key: TranslationKeys, params?: Record<string, string>): string;
```

### Phase 3: React Island への翻訳渡し

#### 方法 A: Props で渡す (推奨)

```astro
---
const locale = getLocale(Astro);
const translations = {
  save: t(locale, "channel.save"),
  cancel: t(locale, "channel.cancel"),
  // ... 必要な翻訳のみ
};
---
<ChannelConfigModal client:load translations={translations} />
```

**メリット**: サーバーサイドで翻訳解決。island のバンドルに辞書全体を含める必要なし  
**デメリット**: 渡す props 数が増加

#### 方法 B: Nano Store で渡す

```typescript
// i18n/stores/locale.ts
import { atom } from "nanostores";

export const $locale = atom<"ja" | "en">("ja");
```

```tsx
// React island 内
import { useStore } from "@nanostores/react";
import { $locale } from "../../i18n/stores/locale";
import { t } from "../../i18n";

function ChannelConfigModal() {
  const locale = useStore($locale);
  return <button>{t(locale, "channel.save")}</button>;
}
```

**メリット**: props 不要
**デメリット**: 辞書全体がクライアントバンドルに含まれる

#### 推奨

小さい island → **方法 A** (props)
大きい island (ChannelConfigModal 等) → **方法 B** (Nano Store) — 翻訳キーが多すぎて props では管理困難

### Phase 4: URL ベースルーティング (将来)

現在はセッションベース (`routing: "manual"`)。URL ベース (`/ja/dashboard`, `/en/dashboard`) への移行は大きな変更のため、別フェーズで検討。

**移行メリット**:

- SEO: 各言語版が個別 URL → Google のインデックスが適切に機能する
- シェアリング: URL にロケール情報が含まれる
- SSR: セッション不要でキャッシュ可能

**移行コスト**:

- 全ルーティングの変更
- ミドルウェアの変更
- リダイレクトルールの追加
- 既存ブックマークの互換性

```typescript
// 将来の astro.config.ts
i18n: {
  defaultLocale: "ja",
  locales: ["ja", "en"],
  routing: {
    prefixDefaultLocale: false, // /dashboard = ja, /en/dashboard = en
  },
}
```

## 翻訳品質チェックリスト

- [ ] 全ての UI テキストが dict.ts に定義されている (ハードコード文字列なし)
- [ ] ja と en の全キーが一致している (欠落キーなし)
- [ ] 動的パラメータ (`{param}`) が正しく設定されている
- [ ] 日付・数値のフォーマットがロケール対応 (`Intl.DateTimeFormat`, `Intl.NumberFormat`)
- [ ] `<html lang>` がロケールに連動している
- [ ] メタタグ (title, description) が翻訳されている
- [ ] `hreflang` alternate リンクが設定されている
- [ ] エラーメッセージが翻訳されている
