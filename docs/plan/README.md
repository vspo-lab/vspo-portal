# Feature Plan (Spec-Driven Development)

Feature development specifications are consolidated here.
The spec is the Single Source of Truth; verbal decisions are invalid.

## Principle: Spec Update -> Implementation

1. When a spec change occurs, first update the document in `docs/plan/<feature>/`
2. Only after the document is updated, modify the code
3. Verbal or chat agreements are not specs. Only what is written is the spec

## Directory Structure

```
docs/plan/
+-- README.md                    # This document
+-- <feature-name>/              # Directory per feature
    +-- 00_OVERVIEW.md           # Feature overview, purpose, scope
    +-- 01_DOMAIN_MODEL.md       # Entity changes, business rules
    +-- 02_DATA_ACCESS.md        # Repository and DB changes
    +-- 03_USECASE.md            # UseCase layer changes
    +-- 04_API_INTERFACE.md      # API endpoint specifications
    +-- 05_FRONTEND.md           # Frontend UI specifications
    +-- CHECKLIST.md             # Implementation checklist (phases and session notes)
```

Files (`.md`) auto-generated by Claude Code's plan mode are also saved in this directory.
Feature directories and auto-generated files have different name formats, so they do not conflict.

## Workflow

### Step 1: Spec Generation (`/plan-feature`)

Create structured specification documents from ambiguous requirements.

- Interview requirements and write specs in the file structure above
- Specs correspond to Clean Architecture layers (Domain -> Data Access -> UseCase -> Interface)
- Mark unresolved items as `TBD` and record next actions

### Step 2: Checklist Generation (`/init-impl`)

Read the specs and generate a phased implementation checklist.

- Implementation order is bottom-up: Domain -> Data Access -> UseCase -> API -> Frontend
- Each phase includes goals, checklists, test commands, and session notes
- Session notes record Done / Next / Risks & TODO

### Step 3: Phase Execution

Implement according to the checklist, progressing phase by phase.

- Initialize session notes at the start of each phase
- Check off completed tasks
- If a spec change is needed, update the spec document first, then modify the code

## Spec File Overview

| File | Content | Corresponding Layer |
| --- | --- | --- |
| `00_OVERVIEW.md` | Feature purpose, background, scope, success criteria | - |
| `01_DOMAIN_MODEL.md` | New/changed entities, business rules, Zod Schema | Domain |
| `02_DATA_ACCESS.md` | Table changes, repository changes, migrations | Infrastructure |
| `03_USECASE.md` | UseCase definitions, inputs/outputs, error cases | UseCase |
| `04_API_INTERFACE.md` | Endpoints, request/response, authentication | Infrastructure (HTTP) |
| `05_FRONTEND.md` | Page structure, components, state management | Frontend |
| `CHECKLIST.md` | Phased implementation checklist, session notes | - |

## How to Write Session Notes

Always record the following 3 items in each phase's Session Notes:

```markdown
### Session Notes
YYYY-MM-DD
- Done: What was completed
- Next: What to do next
- Risks/TODO: Unresolved issues
```

Use these 3 items to restore context when sessions span multiple days or when resuming a session.
